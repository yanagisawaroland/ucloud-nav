etcd 是 K8s 的大脑：所有 K8s 对象（Pod/Service/Deployment等）都存储在 etcd 中。API Server 是唯一能读写 etcd 的组件，其他组件通过 Watch 机制监听变化。这种设计实现了组件解耦：每个控制器只需关注自己感兴趣的资源变化，无需轮询，高效且可靠。

Watch vs 轮询对比
❌ 轮询：每秒发请求
  浪费网络，延迟高
✅ Watch：长连接推送
  即时响应，高效
Watch 机制原理：基于 HTTP/2 长连接，客户端发送 GET /api/v1/pods?watch=true&resourceVersion=xxx，etcd 保持连接，有变化时立即推送 JSON 事件流。API Server 在内存中维护 Watch Cache，减少对 etcd 的直接查询压力。
控制器调谐循环（Reconcile Loop）
控制器核心逻辑：期望状态 vs 实际状态
func Reconcile(ctx, req) {
  // 1. 从 etcd 获取期望状态
  desired := getFromEtcd(req)
  
  // 2. 获取实际状态
  actual := getActualState()
  
  // 3. 计算差异并修正
  if desired != actual {
    reconcile(desired, actual)
  }
}
🔄 幂等性设计
无论 Reconcile 执行多少次，结果都一样。控制器只关注"现在应该是什么状态"，不关注"发生了什么事件"。

控制器调谐循环（Reconcile Loop）：K8s 所有控制器都遵循同一个模式：获取期望状态 → 获取实际状态 → 计算差异 → 执行操作使实际状态趋近期望状态。这个过程是持续循环的，即使发生网络分区或组件重启，最终都会收敛到正确状态（最终一致性）。
