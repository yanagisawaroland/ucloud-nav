滚动更新（Rolling Update）：K8s Deployment 默认策略。按照 maxUnavailable（最多不可用Pod数）和 maxSurge（最多额外Pod数）逐步替换旧 Pod。过程中新旧版本同时对外提供服务，用户请求可能打到任意版本。适合对版本一致性要求不高的场景。

蓝绿部署（Blue-Green）：维护两套完整的生产环境（蓝=当前版本，绿=新版本），切换时只修改 Service 的 selector，流量瞬间从蓝切到绿。出问题可立即切回蓝（秒级回滚）。缺点：需要双倍资源，且切换瞬间仍有少量请求丢失。

金丝雀发布（Canary）：先将少量流量（如5%/20%）切到新版本，通过 Analysis（Prometheus 指标）自动验证新版本质量，成功后逐步放量直到 100%。出问题自动回滚。Istio 提供精确的权重控制，1个Pod也可以承接任意比例流量，不需要按副本数比例。
三种发布策略全面对比
对比项 🔵 滚动更新 🟢 蓝绿部署 🟡 金丝雀发布
服务中断 无（滚动替换） 极短（切换瞬间） 无
回滚速度 较慢（需重新滚动） 秒级（切回蓝） 自动回滚
额外资源 少（maxSurge） 翻倍（两套环境） 极少（1个canary Pod）
流量控制精度 按副本数比例 只能100%切换 任意精度（1%）
版本共存 短暂共存 完整共存 长期共存
复杂度 低（K8s原生） 中（需双环境） 高（需Istio/Rollouts）
质量验证 无自动验证 手动测试绿环境 Analysis自动验证
适用场景 简单业务，快速迭代 高可用，快速切换 大流量，精细控制

选型决策树
Q1: 是否需要精确流量控制？
是 → 金丝雀（需要 Istio）
否 → Q2
Q2: 是否需要秒级回滚能力？
是 → 蓝绿部署（资源充足时）
否 → Q3
Q3: 对发布复杂度要求低？
是 → 滚动更新（K8s默认，够用就行）
实际使用建议：中小项目用滚动更新即可，K8s 原生支持，无需额外组件。追求快速回滚且资源充足时用蓝绿部署。大流量高可用的生产服务用金丝雀发布，配合 Istio + Argo Rollouts + Prometheus Analysis，实现全自动灰度发布和质量验证。三种策略不互斥，可以根据不同服务的重要程度分别采用。
