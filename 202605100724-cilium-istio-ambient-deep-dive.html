🔥 Cilium + Istio + Ambient Mesh 深度事故复盘

💥 一、最终根因（不是权限问题，是“控制面污染”）

核心误判点：

  • ❌ 一开始以为是 RBAC Unauthorized
  • ❌ 实际是 istiod 多版本 ReplicaSet 混存
  • ❌ Helm upgrade + rollout restart 叠加导致控制面发散
failed to list *v1.ConfigMap: Unauthorized
Readiness probe failed: HTTP 503
discovery is not ready

真正根因：

  • ✔ istiod 多版本 RS 同时存在
  • ✔ webhook / CRD / configmap 版本不一致
  • ✔ SA/token 被不同 revision pod 复用
  • ✔ controller cache 无法收敛

⏱ 二、关键故障时间线（真实发生顺序）

T1  istiod 启动成功（1/1）
T2  Helm upgrade 触发新 revision
T3  多 RS 共存（6c66 / 79c4 / 6f4f / 77b9）
T4  webhook 未同步更新
T5  ConfigMap Unauthorized
T6  readiness probe 503
T7  HPA metrics fail
T8  cluster 状态发散

⚠️ 三、真实报错拆解（不是表面现象）

1️⃣ Unauthorized（误导性最强）

failed to list *v1.ConfigMap: Unauthorized
真实含义：
  • 不是 RBAC 没权限
  • 是 istiod 使用的 token 与 API server 认证链不一致
  • 来自“旧 Pod + 新 SA + 混合 webhook”
---

2️⃣ readiness probe 503

HTTP probe failed with statuscode: 503
真实原因：
  • istiod discovery cache 未 ready
  • config store 未 sync
  • CRD + webhook 未统一版本
---

3️⃣ discovery is not ready

  • 控制面未完成 ConfigMap -> xDS pipeline
  • istiod 无法构建 pilot cache

⚡ 四、Cilium 层问题

  • eBPF + kube-proxy replacement 环境稳定性依赖 CNI init 顺序
  • cilium-envoy 重启影响 node agent sync
  • kubelet sandbox create failure（CNI 未 ready）
FailedCreatePodSandBox
failed to setup network for sandbox

🔥 五、Istio 核心坑

  • 多 revision istiod（最大问题）
  • Helm upgrade 未 cleanup old RS
  • webhook config 未同步升级
  • ConfigMap / SA / CRD 版本混合

🌐 六、Ambient Mesh 额外风险点

  • ztunnel 强依赖 node network stable
  • waypoint gateway 依赖 control plane 完全收敛
  • 任何 istiod 不 ready → ambient 全面失效

🛠 七、最终修复路径（你这次真正成功的关键）

✔ 删除所有旧 RS
✔ 清理 webhook
✔ Helm 强制统一版本 1.27.0
✔ 重新生成唯一 istiod pod
✔ 触发 clean reconciliation

最终效果：

  • ✔ 单 istiod RS
  • ✔ 1/1 Running
  • ✔ discovery ready
  • ✔ API access 正常
