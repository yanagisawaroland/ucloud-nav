Percona XtraDB Cluster 与其他 MySQL 集群逻辑关系及流程动态图

Percona XtraDB Cluster（PXC）
应用 / 客户端 ⇄ 任意 PXC 节点
1. 应用可以连接到任意一个 PXC 节点进行读写。
2. 写入通过 Galera 协议在所有节点间同步复制（同步复制 / 组复制）。
3. 所有节点数据保持强一致，任一节点故障时，其他节点仍可提供服务。
4. 通常配合负载均衡（如 HAProxy / ProxySQL）实现高可用与流量分发。
传统主从 / 其他 MySQL 集群
应用 / 客户端 ⇄ 主库（Primary）
从库 1 ← 复制日志 → 从库 2
1. 所有写入集中到主库，主库将 binlog 异步复制到从库。
2. 从库通常只提供读服务，存在复制延迟，读写可能短暂不一致。
3. 主库故障时，需要手动或借助工具进行主从切换，存在故障窗口。
4. 适合读多写少、对强一致性要求不那么极端的场景。
对比视图
PXC：多主写入、同步复制、强一致、高可用，适合对一致性要求高的核心业务。
传统集群：单主多从、异步复制、可能存在延迟，架构相对简单，适合读多写少场景。
共同点：都通过多节点提升可用性与扩展能力，但在一致性、复杂度和运维方式上有明显差异。
