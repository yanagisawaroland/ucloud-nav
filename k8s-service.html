ClusterIP 工作原理：kube-proxy 在每个节点上配置 iptables/ipvs 规则，将访问 ClusterIP:Port 的流量 DNAT 到后端 Pod IP。ClusterIP 是虚拟 IP，不对应任何网卡，只存在于 iptables 规则中。Pod 通过 DNS（CoreDNS）解析 Service 名称得到 ClusterIP。

NodePort 工作原理：在 ClusterIP 基础上，kube-proxy 在每个节点上额外监听一个固定端口（NodePort，范围30000-32767）。访问任意节点的 NodePort，都会被转发到对应 Service 的 Pod。注意：即使 Pod 不在该节点上，流量也能正确路由（跨节点转发）。

MetalLB 工作模式
Layer2（ARP）：VIP 绑定一个节点，该节点响应 ARP
BGP：所有节点通告 VIP 路由，真正多活
B环境：Layer2 模式（ipvs+strictARP）
LoadBalancer 工作原理：在 NodePort 基础上，由云厂商（AWS/GCP/阿里云）或 MetalLB 分配一个外部 IP（VIP）。外部流量打到 VIP，由 LB 转发到各节点 NodePort，再路由到 Pod。k3s 环境用 MetalLB 实现裸金属 LoadBalancer，B 环境 VIP 为 10.20.50.50。
ExternalName — 集群内访问外部服务
📦 集群内 Pod
访问 mysql-svc.default.svc.cluster.local
↓ CoreDNS CNAME 解析
🔵 Service（ExternalName）
mysql-svc → CNAME → db.example.com
↓ DNS 解析到外部地址
🌍 外部服务
db.example.com（RDS/外部MySQL等）
# ExternalName Service YAML
apiVersion: v1
kind: Service
spec:
  type: ExternalName
  externalName: db.example.com
服务信息
类型ExternalName
ClusterIPNone
实现DNS CNAME
kube-proxy不参与
适用场景
✅ 访问外部数据库（RDS）
✅ 访问第三方 API
✅ 迁移期间桥接内外服务
⚠️ 纯 DNS，无负载均衡
与其他类型对比
无 ClusterIP，无 iptables 规则
CoreDNS 返回 CNAME 记录
Pod 自行解析外部域名
ExternalName 工作原理：不创建 ClusterIP，不配置 iptables 规则。CoreDNS 为该 Service 返回 CNAME 记录，指向 externalName 字段的域名。Pod 访问 Service 名称时，DNS 解析到外部域名，再由外部 DNS 解析到实际 IP。常用于将外部数据库/服务映射为集群内的 Service 名称，方便迁移和切换。
四种 Service 类型对比总结
类型外部访问实现方式适用场景我们的环境
ClusterIP ❌ 仅集群内 iptables DNAT 微服务内部通信、数据库 所有环境默认使用
NodePort ⚠️ 节点IP+端口 iptables + 节点端口 开发测试、Kind环境 D环境访问ArgoCD
LoadBalancer ✅ 独立VIP 云LB / MetalLB 生产环境对外暴露 B环境 MetalLB 10.20.50.50
ExternalName - DNS CNAME 访问外部服务 对接外部RDS等

选型建议：集群内部服务间调用用 ClusterIP（默认）；开发测试临时暴露用 NodePort；生产环境对外暴露用 LoadBalancer（云环境自动分配/裸金属用MetalLB）；需要访问外部服务用 ExternalName。Istio IngressGateway 本质上是一个 LoadBalancer 类型的 Service，B 环境的 VIP 10.20.50.50 就是 Istio Gateway 的 LoadBalancer IP。
