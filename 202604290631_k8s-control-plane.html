各组件职责详解
组件职责说明
kubectl命令行工具，用户通过它向 API Server 发送 RESTful 请求（如 kubectl apply/get/delete），是人与集群交互的主要入口。
API Server集群的统一网关和通信枢纽。所有组件（包括 kubectl、Scheduler、Controller Manager、kubelet 等）都只能通过 API Server 进行通信。负责认证(Authentication)、授权(Authorization)、准入控制(Admission Control)，并将数据持久化到 etcd。
etcd分布式键值存储数据库，保存集群所有状态数据（Pod、Service、ConfigMap、Secret 等）。只有 API Server 能直接读写 etcd，其他组件必须通过 API Server 间接访问。采用 Raft 协议保证一致性。
Scheduler调度器，通过 Watch 机制监听 API Server，发现 nodeName 为空的 Pod 后触发调度。先执行过滤(Filtering)排除不满足条件的节点（资源不足、亲和性不匹配等），再执行打分(Scoring)选出最优节点，最后通过 API Server 更新 Pod 的 nodeName 字段。
Controller Manager控制器管理器，内部运行多个控制器：Deployment Controller、ReplicaSet Controller、Node Controller、Endpoint Controller、ServiceAccount Controller 等。每个控制器通过 Watch 机制监听资源变化，执行 Reconcile 循环，确保集群实际状态始终趋向期望状态。
kubelet运行在每个 Worker 节点上的代理。Watch API Server 获取调度到本节点的 Pod 信息，调用容器运行时（containerd/CRI-O）创建和管理容器。同时负责执行健康检查（Liveness/Readiness Probe）、挂载存储卷、上报节点和 Pod 状态。
kube-proxy运行在每个节点上的网络代理。Watch API Server 中 Service 和 Endpoint 对象的变化，动态维护节点上的 iptables 或 IPVS 规则，实现 Service ClusterIP 到后端 Pod IP 的流量转发和负载均衡。
Cloud Controller Manager云控制器管理器（仅云环境），与云厂商 API 交互。管理 LoadBalancer 类型 Service 的云负载均衡器创建、Node 对象与云实例的映射、路由配置等。将云相关逻辑从核心控制器中解耦。
完整流程说明（以创建 Deployment 为例）：
  1. 用户提交请求：用户执行 kubectl apply -f deployment.yaml，kubectl 将 YAML 转换为 JSON 并发送 POST 请求到 API Server。
  2. API Server 认证授权：API Server 收到请求后，依次执行认证（验证用户身份）、授权（检查 RBAC 权限）、准入控制（执行 MutatingWebhook/ValidatingWebhook），全部通过后将 Deployment 对象写入 etcd。
  3. Controller Manager 响应：Deployment Controller 通过 Watch 机制发现新的 Deployment 对象，创建对应的 ReplicaSet；ReplicaSet Controller 随后创建指定数量的 Pod 对象（此时 Pod 的 nodeName 为空）。
  4. Scheduler 调度 Pod：Scheduler 通过 Watch 发现 nodeName 为空的 Pod，执行过滤算法（检查节点资源、污点容忍、亲和性等）和打分算法（评估资源利用率、数据局部性等），选出最优节点，将 Pod 绑定信息（nodeName）通过 API Server 写入 etcd。
  5. kubelet 创建容器：目标节点的 kubelet 通过 Watch 发现有 Pod 调度到本节点，调用容器运行时拉取镜像、创建容器、配置网络（调用 CNI 插件）、挂载存储卷，启动容器并开始执行健康检查。
  6. kubelet 上报状态：kubelet 持续向 API Server 汇报节点状态（CPU/内存/磁盘）和 Pod 状态（Running/Failed/Succeeded），API Server 将状态更新写入 etcd。
  7. Endpoint Controller 更新：当 Pod 变为 Ready 状态后，Endpoint Controller 将 Pod IP 添加到对应 Service 的 Endpoints 对象中。
  8. kube-proxy 更新规则：kube-proxy Watch 到 Service/Endpoints 变化后，更新本节点的 iptables/IPVS 规则，使流量能正确转发到新的 Pod。
  9. Cloud Controller 联动：如果 Service 类型为 LoadBalancer，Cloud Controller Manager 会调用云厂商 API 创建/更新外部负载均衡器，将外部流量引入集群。
