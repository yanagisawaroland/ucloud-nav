K3s + Cilium + Istio Ambient 安装问题总结

环境： K3s v1.35.4 | Cilium v1.19.1 | Istio Ambient v1.29.0

日期： 2026-05-12

问题一：istio-ingressgateway 和 ztunnel 卡在 ContainerCreating

现象

istio-system   istio-ingressgateway-7876fb579-2nlj8   0/1   ContainerCreating   0   3h11m
istio-system   ztunnel-fr8rc                          0/1   ContainerCreating   0   3h11m

错误信息：

Failed to create pod sandbox: rpc error: code = DeadlineExceeded
desc = stream terminated by RST_STREAM with error code: CANCEL

根本原因

Cilium 安装时未启用 ambient 模式支持（缺少相关参数），导致 05-cilium.conflist 中的 istio-cni 插件配置了错误的 enablement_selectors。

istio-cni 插件要求 pod 必须带有标签 istio.io/dataplane-mode: ambient 才会处理，但 ztunnel、istio-ingressgateway 等系统组件本身没有这个标签，且 istio-system 命名空间未被加入排除列表。

istio-cni 日志中可以看到明确报错：

"not a CNI pod, podName: ztunnel-fr8rc, podNamespace: istio-system"
"not a CNI pod, podName: istio-ingressgateway-7876fb579-2nlj8, podNamespace: istio-system"

istio-cni 跳过处理这些 pod → sandbox 创建流程卡死 → DeadlineExceeded。

master91 上的 ztunnel（ztunnel-qj8xv）能正常运行，是因为它在问题出现之前就已启动成功，属于时序侥幸。

解决方法

在 istio-ambient.yaml 中为 istio-system 命名空间加入 CNI 排除列表：

apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
metadata:
  namespace: istio-system
spec:
  hub: harbor.devops.local:7581/devops91
  tag: 1.29.0
  profile: ambient
  components:
    ingressGateways:
    - name: istio-ingressgateway
      enabled: true
    pilot:
      k8s:
        resources:
          requests:
            memory: 200Mi
            cpu: 100m
  values:
    global:
      proxy:
        resources:
          requests:
            memory: 64Mi
            cpu: 10m
    cni:
      cniBinDir: /var/lib/rancher/k3s/data/current/bin
      cniConfDir: /etc/cni/net.d
      excludeNamespaces:
        - kube-system
        - istio-system        # ← 关键：排除 istio-system 自身

重新安装：

istioctl install -f ~/istio-ambient.yaml -y
kubectl rollout restart daemonset/istio-cni-node -n istio-system
kubectl rollout status daemonset/istio-cni-node -n istio-system

验证

kubectl get pod -n istio-system -o wide
# 所有 pod 应为 1/1 Running

问题二：Cilium status 报 localhost:8080 错误

现象

执行 cilium status 时报错：

Get "http://localhost:8080/apis/apps/v1/...": dial tcp 127.0.0.1:8080: connect: connection refused

根本原因

cilium CLI 命令没有读取 k3s 的 kubeconfig，默认去找 localhost:8080（标准 Kubernetes 默认地址）。

解决方法

执行任何 cilium CLI 命令前设置环境变量：

export KUBECONFIG=/etc/rancher/k3s/k3s.yaml
cilium status

建议写入 ~/.bashrc 永久生效：

echo 'export KUBECONFIG=/etc/rancher/k3s/k3s.yaml' >> ~/.bashrc
source ~/.bashrc

经验总结

问题 根因 关键排查命令
pod sandbox ContainerCreating istio-cni 跳过 istio-system 命名空间的 pod kubectl logs istio-cni-node-xxx | grep "not a CNI pod"
cilium status 报 8080 错误 KUBECONFIG 未设置 export KUBECONFIG=/etc/rancher/k3s/k3s.yaml

注意事项

  1. Istio ambient 模式下，istio-system 命名空间必须加入 CNI excludeNamespaces，否则 istio 自身的系统组件无法创建 sandbox。
  2. K3s 环境下所有需要访问 apiserver 的工具（helm、cilium CLI 等）都需要设置 KUBECONFIG=/etc/rancher/k3s/k3s.yaml。
  3. Cilium 与 Istio ambient 的 CNI 链式调用要求 cni.exclusive=false，缺少此参数会导致 CNI 配置被覆盖。
